## 1. 6-DOF 状态反馈迁移

### 背景

旧系统 pos/vel/thr 只传 4 元素 `[x, y, z, yaw]`，roll/pitch 在 MCU 固件层被截断。position 节点的射线方向计算只用 `R(0, 0, yaw)` 旋转，roll/pitch 误差导致 3D 定位偏差。

### 改动

| 文件 | 改动 |
|------|------|
| `sim_bridge.py` | `_odom_cb` 提取完整 RPY；`_publish_pos/vel/thr` 发布 6 元素 |
| `coordinate.py` | `from_zit6_array` 兼容 4/6 元素；`to_zit6_array` 输出 6 元素 |
| `basic_motion.py` | `_pos_cb` / `_vel_cb` 兼容 4/6 元素；PoseInfo 新增 `robot_roll` / `robot_pitch` |
| `position.py` | `_euler_to_rotation_matrix(roll, pitch, yaw)` 取代 `R(0, 0, yaw)` |

**兼容策略**：`len(msg.data) >= 6` 检查，实车 MCU 未升级时 roll/pitch=0，退化为 2D 行为。

## 2. 时间戳同步

### 问题

旧链路中 PoseInfo 的时间戳是发布时刻（`get_clock().now()`），与 ZIT6 位姿的采集时刻有数十毫秒延迟。position 节点用当前 PoseInfo 处理历史检测帧，运动中的物体定位有偏差。

### 方案

```
ZIT6 pos 到达 → _pos_cb 记录 now() → PoseInfo.stamp = 接收时刻
                                         ↓
position._pose_buffer (deque, 2s/150条)  ←  PoseInfo 入队
                                         ↓
检测帧到达 (header.stamp) → _lookup_pose() → bisect 二分查找最近位姿
                                         ↓
                                    时间对齐的姿态 → 射线计算
```

| 文件 | 改动 |
|------|------|
| `basic_motion.py` | `_pos_cb` 记录 `self._pose_stamp`；`_publish_pose_info` 使用该时间戳 |
| `position.py` | 新增 `_pose_buffer: deque`；`_lookup_pose()` 二分查找；`_process_detection` 使用时间对齐位姿 |

## 3. position 节点算法优化（7 项）

参考外部代码审查文档 `position修改建议.md`，排除多实例跟踪和时间清理：

| # | 改进 | 实现 |
|---|------|------|
| 1 | 文档注释修正 | "RANSAC" → "median filtering" |
| 2 | 高斯像素加权 | `weight = exp(-0.5 * (dist/sigma)²)`，边缘像素降权（min 0.1） |
| 3 | WLS 3D 交汇 | 加权最小二乘 `A += w*P, b += w*(P@o)`，替代简单均值 |
| 4 | 成对交汇改进 | 对权重 = w1×w2，粗筛中位数，离群剔除 >1.5m，加权平均 inliers |
| 5 | 加权裁剪 + 除零安全 | `cos_sim / (dist + 1e-3)`；`score = score / wi` 惩罚边缘射线 |
| 6 | EMA 位置平滑 | `old_pos * 0.7 + position * 0.3`（α=0.3），仅对非新物体 |
| 7 | 二分查找优化 | `bisect.bisect_left` O(log n)，替代 O(n) 线性扫描 |

### 新增 ROS 参数

| 参数 | 默认 | 说明 |
|------|------|------|
| `pose_buffer_duration` | `2.0` | 位姿历史保留时长（秒） |
| `max_history` | `30` | 每个物体的最大射线数 |

## 4. 多实例数据关联

### 问题

旧 key 为 `(class_id, camera_pair)`，同画面多个同类物体（两根箭头）射线混入同一 history，交汇出幽灵定位点。

### 方案

新增 `instance_id` 字段，key 变为 `(class_id, camera_pair, instance_id)`。

**类别分级**：

```python
MULTI_INSTANCE_CLASSES = {3, 5, 6}   # arrow, triangle, square
```

| 物体 | class_id | 行为 |
|------|----------|------|
| yellow/red/green_sector | 0,1,2 | 始终 instance 0 |
| **arrow** | **3** | 角度匹配，可有多实例 |
| start | 4 | 始终 instance 0 |
| **triangle** | **5** | 角度匹配，可有多实例 |
| **square** | **6** | 角度匹配，可有多实例 |
| basket | 7 | 始终 instance 0 |
| aruco_tag | 8 | 始终 instance 0 |

**四阶段关联策略**（仅多实例类）：

```
Phase 1: 无任何实例 → instance 0（冷启动）
Phase 2: 3D 位置角度匹配（cos > cos(50°)，即夹角 < 50°）
Phase 3: 填充不完整实例（有射线没位置的优先，射线方向匹配或最少优先）
Phase 4: 所有实例满 + 都不匹配 → 真正的新物体
```

### 改动文件

| 文件 | 改动 |
|------|------|
| `ObjectPosition.msg` | 新增 `uint32 instance_id` |
| `position.py` | `_associate_ray()` 四阶段关联；`_broadcast_objects` 写入 `instance_id` |
| `position.py` | 新增 `MULTI_INSTANCE_CLASSES` 常量；新增 `angle_match_threshold` 参数（默认 50°） |

### 下游影响：零改动

`navigator.py`（只取 `world_x/y`）和 `task_runner.py`（只存引用）不受影响。
