## Scene 文件格式整理

### 问题

`xunyun_fixed.scn` 第 498 行的 `<world_transform rpy="0.0 0.0 0.0"` 写死入水朝向，每次改方向要改文件本身。通过 `include` 引用时没法按需传参。

### 改动

`xunyun_fixed.scn`：

- 新增 `<arg name="robot_orientation" default="0.0 0.0 0.0"/>`
- `<world_transform rpy="$(arg robot_orientation)" xyz="$(arg robot_position)"/>`

现在引用时可以：

```xml
<include file="xunyun_fixed.scn">
    <arg name="robot_name" value="XUNYUN"/>
    <arg name="robot_position" value="10.5 -2.8 0.3"/>
    <arg name="robot_orientation" value="0.0 0.0 1.57"/>  <!-- 朝东 -->
</include>
```

## 图片 Topic 全链路审计

### 结论

```
Stonefish GPU  ─4×─→  sim_bridge  ─6×─→  vision  ─4×det+4×img─→  position
  /sim/*/image_color      /auv/*/stitched     /perception/detection/*
                            /auv/*/left|right
```

**冗余发布**（发无人收）：

| Topic                             | 发布者     | 为什么无用                           |
| --------------------------------- | ---------- | ------------------------------------ |
| `/auv/front_cam/left`           | sim_bridge | vision 只消费`*stitched`           |
| `/auv/front_cam/right`          | sim_bridge | 同上                                 |
| `/auv/down_cam/left`            | sim_bridge | 同上                                 |
| `/auv/down_cam/right`           | sim_bridge | 同上                                 |
| `/perception/image/front_left`  | vision     | 仅`publish_images=true` 发，无人收 |
| `/perception/image/front_right` | vision     | 同上                                 |
| `/perception/image/down_left`   | vision     | 同上                                 |
| `/perception/image/down_right`  | vision     | 同上                                 |

**GPU 出图条件**：`stonefish_simulator`（GPU 版，`ROS2GraphicalSimulationApp`）需要 X Display / GL 窗口才能渲染相机传感器。无图形环境时使用 `xvfb-run` 启动。

## MCU 固件审查

对 `/home/origin/AUV_zit6_cmake/` 的审查结果：

- `NavState.pos_world` 是 `std::array<float, 6>`，索引 `[0]=X [1]=Y [2]=Z [3]=Roll [4]=Pitch [5]=Yaw`
- `MicroRosPublisher.cpp` 分配 `capacity=4`、`size=4`，只发布 [0][1][2][5]，**丢弃了 roll 和 pitch**
- 同样的模式也出现在速度反馈（`vel_body` 跳过了 index 3/4）和推力反馈

**结论**：MCU 固件内部有 roll/pitch，协议层截断了。后续需要：

1. MCU 侧：`size=capacity=6`，补发 roll_rad / pitch_rad
2. 上位机：兼容 4 元素旧格式 / 6 元素新格式

## 时间戳同步方案

已完成详细计划（`/home/origin/.claude/plans/jaunty-stirring-meerkat.md`），涵盖 7 个文件：

| 文件                   | 改动要点                                                         |
| ---------------------- | ---------------------------------------------------------------- |
| `PoseInfo.msg`       | +robot_roll / robot_pitch                                        |
| `ObjectPosition.msg` | +detection_stamp                                                 |
| `sim_bridge.py`      | `_odom_cb` 保存全姿态（roll/pitch），ZIT6 协议 4→6 元素       |
| `coordinate.py`      | `from_zit6_array` 兼容 4/6 元素；`to_zit6_array` 输出 6 元素 |
| `basic_motion.py`    | `_pos_cb` 兼容 4/6 格式；PoseInfo 附带 roll/pitch              |
| `position.py`        | **核心**：二分查姿态、全 3D 旋转矩阵、射线/物体带时间戳    |
| `vision.py`          | DetectionArray.header.stamp 已在传，无需改动                     |

**设计要点**：

- 30Hz 位姿历史 + `bisect` 二分查找（O(log n)）
- 30Hz × 20s = 600 条 buffer，启动初期回退到当前值
- 全 3D 旋转矩阵 `R(roll, pitch, yaw)` 取代原来的 `R(0, 0, yaw)`
- 实车 MCU 未升级时 roll/pitch=0，旋转矩阵退化为 2D，行为不变

## 其他修复

### NumPy 降级

ROS 2 Jazzy 的 cv_bridge 编译时链接了 NumPy 1.x，运行环境装了 NumPy 2.5.1 导致崩溃：

```bash
pip install 'numpy<2' --break-system-packages
```

降级到 1.26.4 后 cv_bridge 正常工作。opencv-python 4.13 虽标记需要 numpy≥2，但运行时兼容。

### CLAUDE.md 修正

`Layer 0` 描述从 "NO GPU mode" 修正为 "GPU mode"。

### YOLO 模型路径更新

默认模型从 `SAUVC_sim52.pt` 更新为 `WUURC2026_sim_719.pt`（5.4MB）。
注意：修改后需重新编译才能生效（或使用 `--symlink-install`）。
